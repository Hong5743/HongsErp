# PRD: 홍스 의류 생산 관리 ERP 시스템

**Label:** `ready-for-agent`

---

## Problem Statement

의류 생산 에이전트 회사(홍스)는 브랜드(발주처)와 공장(생산처) 사이에서 오더를 중간 관리한다. 현재 모든 업무를 엑셀로 관리하고 있어 다음 문제가 발생한다:

- 오더가 현재 어떤 단계에 있는지 한눈에 파악하기 어렵다.
- 납기일 임박을 놓치는 경우가 발생한다.
- 샘플(핏샘플·PP샘플) 진행 상태, 부자재 투입 여부, 정산 정보가 파일마다 흩어져 있다.
- 연차 신청과 결재가 구두·메신저로 이루어져 기록이 남지 않는다.
- 인보이스·세금계산서 작성이 수작업이라 오류가 생기기 쉽다.

---

## Solution

Spring Boot + MySQL + Redis 백엔드, Next.js 프론트엔드(반응형 웹, 한국어·영어 지원), AWS 배포로 구성된 사내 ERP를 구축한다.

핵심 가치는 **오더 상태 대시보드** — 모든 오더의 진행 단계를 한 화면에서 파악할 수 있게 한다. 여기에 샘플 관리, 부자재 관리, 정산·PDF 문서 출력, 사원 관리·연차 결재, 내부 알림을 통합한다.

---

## User Stories

### 인증 / 권한

1. 관리자로서, 이메일과 비밀번호로 로그인할 수 있다. 그래야 내 권한 범위 안에서 ERP를 사용할 수 있다.
2. 일반 직원으로서, 이메일과 비밀번호로 로그인할 수 있다. 그래야 담당 오더와 내 연차를 관리할 수 있다.
3. 관리자로서, 직원 계정을 생성·수정·비활성화할 수 있다. 그래야 퇴사자의 접근을 막을 수 있다.
4. 일반 직원으로서, 정산·수익 정보는 볼 수 없다. 그래야 민감한 재무 정보가 보호된다.
5. 관리자로서, 직원의 역할(관리자/일반)을 변경할 수 있다. 그래야 승진이나 역할 변경에 대응할 수 있다.

### 오더 관리

6. 오더 담당자로서, 새 오더를 등록할 수 있다. 그래야 오더 정보를 ERP에서 일괄 관리할 수 있다.
7. 오더 담당자로서, 오더에 브랜드명·스타일명·스타일번호·컬러·사이즈별 수량·브랜드 요청 납기·공장 납기·담당 공장·브랜드 단가·공장 원가를 입력할 수 있다. 그래야 오더 정보를 한 곳에서 관리할 수 있다.
8. 오더 담당자로서, 오더의 진행 단계를 변경할 수 있다 (오더 접수 → 원단·부자재 소싱 → 공장 배정 → 샘플 진행 → 본 생산 → QC → 선적·납품 → 정산 완료). 그래야 현재 상태를 정확히 기록할 수 있다.
9. 모든 사용자로서, 오더 목록을 단계별·브랜드별·담당자별로 필터링할 수 있다. 그래야 원하는 오더를 빠르게 찾을 수 있다.
10. 모든 사용자로서, 전체 오더의 진행 단계를 칸반 또는 테이블 형태의 대시보드에서 한눈에 볼 수 있다. 그래야 어떤 오더가 어디서 막혀 있는지 즉시 파악할 수 있다.
11. 오더 담당자로서, 오더를 수정·삭제할 수 있다. 그래야 입력 오류를 바로잡을 수 있다.
12. 모든 사용자로서, 오더 상세 페이지에서 샘플 진행·부자재·정산 정보를 모두 볼 수 있다. 그래야 오더와 관련된 모든 정보를 한 곳에서 확인할 수 있다.

### 샘플 관리

13. 오더 담당자로서, 오더별로 핏샘플 진행 상태를 기록할 수 있다 (의뢰 → 수령 → 피팅·수정 요청 → 승인). 그래야 샘플 단계를 놓치지 않을 수 있다.
14. 오더 담당자로서, 오더별로 PP샘플 진행 상태를 기록할 수 있다 (의뢰 → 수령 → 검토 → 승인). 그래야 본 생산 전 최종 확인이 완료됐는지 추적할 수 있다.
15. 오더 담당자로서, 샘플 수령일·승인일·수정 요청 내용을 메/모로 남길 수 있다. 그래야 히스토리를 추적할 수 있다.
16. 관리자로서, 전체 오더의 샘플 진행 현황을 한 화면에서 볼 수 있다. 그래야 지연되고 있는 샘플을 빠르게 파악할 수 있다.

### 부자재 관리

17. 오더 담당자로서, 오더별로 필요한 부자재 목록을 등록할 수 있다 (부자재명·수량·단위). 그래야 무엇이 필요한지 명확히 관리할 수 있다.
18. 오더 담당자로서, 부자재별로 공장 투입 여부를 체크할 수 있다. 그래야 부자재가 빠져 생산이 지연되는 상황을 막을 수 있다.
19. 담당자로서, 부자재 재고 수량을 직접 입력·수정할 수 있다. 그래야 실사 후 재고 현황을 기록할 수 있다.
20. 관리자로서, 재고 수량이 기준치 이하인 부자재 목록을 확인할 수 있다. 그래야 선제적으로 발주할 수 있다.

### 거래처 관리

21. 관리자로서, 브랜드 거래처를 등록·수정·삭제할 수 있다 (회사명·담당자·연락처·주소·결제 조건). 그래야 브랜드 정보를 ERP에서 일관되게 관리할 수 있다.
22. 관리자로서, 공장 거래처를 등록·수정·삭제할 수 있다 (공장명·담당자·연락처·주소·주요 생산 품목). 그래야 공장 정보를 체계적으로 관리할 수 있다.
23. 오더 담당자로서, 오더 등록 시 기존 브랜드·공장 목록에서 선택할 수 있다. 그래야 거래처 정보를 매번 새로 입력하지 않아도 된다.

### 정산 관리

24. 관리자로서, 오더별로 브랜드 수금액·수금일·수금 상태를 기록할 수 있다. 그래야 미수금을 추적할 수 있다.
25. 관리자로서, 오더별로 공장 지급액·지급일·지급 상태를 기록할 수 있다. 그래야 미지급금을 관리할 수 있다.
26. 관리자로서, 오더별 마진(수금액 - 지급액 - 기타 비용)을 자동으로 계산해서 볼 수 있다. 그래야 수익성을 즉시 파악할 수 있다.
27. 관리자로서, 전체 오더의 정산 현황(총 수금·총 지급·총 마진)을 기간별로 조회할 수 있다. 그래야 월간·분기 실적을 파악할 수 있다.

### 문서 출력 (PDF)

28. 관리자로서, 오더 정보를 바탕으로 세금계산서를 PDF로 출력할 수 있다 (공급자·공급받는자·품목·금액·세액). 그래야 수작업 없이 문서를 생성할 수 있다.
29. 관리자로서, 해외 브랜드·공장용 인보이스를 영어로 PDF 출력할 수 있다 (Shipper·Consignee·Description·Quantity·Unit Price·Total Amount). 그래야 해외 거래처에 즉시 전달할 수 있다.
30. 관리자로서, 출력 전 미리보기를 볼 수 있다. 그래야 오류를 사전에 확인할 수 있다.

### 사원 관리

31. 관리자로서, 사원 정보를 등록할 수 있다 (이름·이메일·연락처·부서·직급·입사일·역할). 그래야 인사 정보를 ERP에서 관리할 수 있다.
32. 관리자로서, 사원 정보를 수정·비활성화할 수 있다. 그래야 인사 변동에 즉시 대응할 수 있다.
33. 관리자로서, 전체 사원 목록을 조회할 수 있다. 그래야 조직 현황을 파악할 수 있다.
34. 일반 직원으로서, 내 프로필 정보를 조회할 수 있다. 그래야 내 인사 정보를 확인할 수 있다.

### 연차 관리

35. 일반 직원으로서, 연차 신청서를 작성할 수 있다 (신청 날짜·시작일·종료일·사유). 그래야 구두 보고 없이 공식적으로 신청할 수 있다.
36. 일반 직원으로서, 내 연차 신청 목록과 승인 상태를 조회할 수 있다. 그래야 신청 현황을 실시간으로 확인할 수 있다.
37. 일반 직원으로서, 승인 전 연차 신청을 취소할 수 있다. 그래야 일정 변경에 대응할 수 있다.
38. 관리자로서, 대기 중인 연차 신청 목록을 볼 수 있다. 그래야 빠르게 처리할 수 있다.
39. 관리자로서, 연차 신청을 승인 또는 반려할 수 있다 (반려 시 사유 입력). 그래야 결재 기록이 남는다.
40. 일반 직원으로서, 내 잔여 연차 일수를 볼 수 있다. 그래야 연차 계획을 세울 수 있다.
41. 관리자로서, 전체 직원의 연차 사용 현황을 조회할 수 있다. 그래야 인력 계획을 수립할 수 있다.

### 알림

42. 오더 담당자로서, 담당 오더의 공장 납기 3일 전에 내부 알림을 받는다. 그래야 납기를 놓치지 않을 수 있다.
43. 일반 직원으로서, 연차 신청이 승인·반려됐을 때 내부 알림을 받는다. 그래야 결과를 즉시 알 수 있다.
44. 관리자로서, 새 연차 신청이 들어오면 내부 알림을 받는다. 그래야 빠르게 처리할 수 있다.
45. 모든 사용자로서, 읽지 않은 알림 수를 헤더 벨 아이콘에서 확인할 수 있다. 그래야 놓친 알림이 있는지 바로 알 수 있다.
46. 모든 사용자로서, 알림 목록을 열어 읽음 처리할 수 있다. 그래야 알림을 정리할 수 있다.

### 데이터 이전

47. 관리자로서, 기존 엑셀 데이터를 ERP로 일괄 가져올 수 있다 (오더, 거래처, 사원 정보). 그래야 과거 데이터를 잃지 않고 ERP로 전환할 수 있다.

### 다국어

48. 모든 사용자로서, UI 언어를 한국어·영어 중 선택할 수 있다. 그래야 외국인 사용자나 해외 거래처 담당자도 사용할 수 있다.

## Implementation Decisions

### 아키텍처

- **프론트엔드**: Next.js (App Router), TailwindCSS, 반응형 웹
- **백엔드**: Spring Boot 3.x, RESTful API
- **데이터베이스**: MySQL 8.x (운영), H2 (테스트)
- **캐시**: Redis — JWT 토큰 블랙리스트(로그아웃), 세션 관리, 자주 조회되는 오더 목록 캐싱
- **배포**: AWS (EC2 + RDS + ElastiCache)
- **ORM**: JPA (Hibernate) — 도메인 모델링, MyBatis — 복잡한 조회 쿼리

### 인증

- JWT 기반 인증 (Access Token + Refresh Token)
- Access Token은 Redis에 블랙리스트 방식으로 로그아웃 처리
- 역할: `ADMIN`, `EMPLOYEE` 두 가지

### 핵심 도메인 모델 (테이블)

- `employee` — 사원 정보 + 역할
- `brand` — 브랜드 거래처
- `factory` — 공장 거래처
- `order` — 오더 (브랜드·공장 FK, 진행 단계 enum)
- `order_size_quantity` — 오더별 사이즈·수량 (1:N)
- `fit_sample` — 핏샘플 진행 (order FK)
- `pp_sample` — PP샘플 진행 (order FK)
- `accessory` — 부자재 마스터
- `order_accessory` — 오더별 부자재 목록 + 투입 여부 + 재고 수량
- `settlement` — 오더별 정산 (수금·지급·마진)
- `leave_request` — 연차 신청
- `notification` — 내부 알림

### 오더 진행 단계 (Enum)

`ORDER_RECEIVED` → `SOURCING` → `FACTORY_ASSIGNED` → `SAMPLING` → `PRODUCTION` → `QC` → `SHIPPING` → `SETTLED`

### 샘플 진행 단계 (Enum)

핏샘플: `REQUESTED` → `RECEIVED` → `FITTING` → `APPROVED`
PP샘플: `REQUESTED` → `RECEIVED` → `REVIEWING` → `APPROVED`

### PDF 출력

- 백엔드에서 iText 또는 JasperReports로 PDF 생성 후 스트림 반환
- 세금계산서: 한국어, 공급자·공급받는자·품목·금액·세액 포함
- 인보이스: 영어, Shipper·Consignee·Description·Quantity·Unit Price·Total Amount 포함

### 다국어 (i18n)

- 프론트엔드: `next-intl` 라이브러리 사용
- 언어 파일: `ko.json`, `en.json`
- 인보이스 PDF는 항상 영어로 출력

### 알림

- 납기 임박 알림: Spring Scheduler (`@Scheduled`)로 매일 자정 납기 3일 전 오더 체크 후 알림 생성
- 알림은 DB에 저장, 프론트는 폴링 또는 SSE(Server-Sent Events)로 실시간 반영

### 엑셀 데이터 이전

- Apache POI로 엑셀 파일 파싱
- 관리자 전용 `/admin/import` 엔드포인트로 일괄 업로드

### Redis 활용 전략

- JWT 블랙리스트 (로그아웃 처리)
- Refresh Token 저장
- 오더 목록 조회 캐싱 (TTL: 5분)

## Testing Decisions

### 좋은 테스트의 기준

- 구현 세부사항(특정 메서드 호출, 내부 변수)이 아닌 **외부 동작(HTTP 응답, DB 상태)**을 테스트한다.
- 하나의 테스트는 하나의 동작만 검증한다.
- 테스트는 독립적이며 순서에 무관하게 실행될 수 있어야 한다.

### 백엔드 테스트

- **통합 테스트** (`@SpringBootTest` + `MockMvc`): HTTP 엔드포인트를 실제로 호출하여 컨트롤러 → 서비스 → 레포지토리 전체를 검증한다.
- **테스트 DB**: H2 인메모리 DB 사용 (`spring.jpa.hibernate.ddl-auto=create-drop`)
- 테스트 대상 모듈: 오더 API, 사원 API, 연차 API, 정산 API, 샘플 API, 부자재 API, 인증 API
- 각 API의 정상 케이스 + 주요 예외 케이스(권한 없음, 존재하지 않는 리소스) 커버

### 프론트엔드 테스트

- **E2E 테스트** (Playwright): 브라우저를 자동 실행하여 사용자 흐름 전체를 검증한다.
- 백엔드 완성 후 단계적으로 추가한다.
- 우선 대상: 로그인, 오더 등록·상태 변경, 연차 신청·결재 흐름

## Out of Scope

- 실제 전자세금계산서 발행 (국세청 API 연동) — PDF 출력으로 대체
- 실시간 부자재 재고 추적 — 담당자 수동 입력 방식으로 운영
- 외부 PR 트리아지 (GitHub PR 관리)
- 모바일 네이티브 앱 (iOS/Android) — 반응형 웹으로 대체
- 급여 관리, 회계 장부
- 카카오톡·이메일 알림 — 내부 알림으로만 운영
- 다중 회사 멀티테넌시

## Further Notes

- GitHub 원격 저장소 설정 후 이 PRD를 GitHub Issue로 등록하고 `ready-for-agent` 라벨을 적용할 것.
